# 知老 / 元核知老 PRD v1.0-v1.5

## 1. 文档目的

本文档用于记录当前项目的真实开发范围、版本演进、已完成能力、明确不做的内容，以及下一阶段计划。

本 PRD 不展开远期大而全方案，重点避免后续开发跑偏。当前产品仍以“养老机构、社区养老中心、老年公益组织、志愿服务团队的一线关怀工具”为核心。

## 2. 当前产品定位

知老是一套面向养老服务场景的长者关怀辅助系统。

当前阶段的核心目标是：

1. 帮服务人员沉淀长者事实信息。
2. 用知老卡帮助服务人员更好理解长者。
3. 用服务机会提醒后续关怀动作。
4. 用服务记录沉淀真实服务过程。
5. 让机构管理员能用最小成本管理试点数据。

当前产品不是：

- 医疗诊断系统
- 心理诊断系统
- 工单派单系统
- 大屏数据驾驶舱
- 社交平台
- 活动管理系统
- 志愿者荣誉系统
- 硬件设备平台

## 3. 当前核心业务闭环

当前真实业务闭环为：

1. 人工填写长者事实信息。
2. 系统整理为知老卡。
3. 服务人员查看知老卡。
4. 系统根据长者信息和服务记录生成服务机会。
5. 服务人员完成真实服务后填写服务记录。
6. 如果服务记录来自某条服务机会，则保存记录后自动完成该机会。
7. 服务记录反哺知老卡和新的服务机会。

## 4. 当前核心模块

### 4.1 首页

首页用于展示机构空间概览，包括：

- 当前机构
- 在册长者
- 待处理服务机会
- 本月服务记录
- 今日重点长者
- 今日服务机会
- 最近服务记录
- 快捷入口

### 4.2 长者档案

长者档案用于维护长者基础资料和知老卡内容。

当前支持：

- 新增长者
- 编辑长者
- 头像上传
- 图片压缩
- 头像预览
- 本地保存
- 知老卡详情
- 长者归档
- 已归档长者恢复

### 4.3 服务机会

服务机会是系统发现的轻提醒，不是强制工单。

当前状态：

- `pending` 待处理
- `completed` 已完成
- `dismissed` 已关闭

当前规则：

- 服务机会不能直接点击状态变成已完成。
- 点击“查看”进入服务机会详情。
- 详情中可查看提醒原因、对应长者、建议动作。
- 可查看知老卡。
- 可去记录服务。
- 可关闭机会，关闭必须选择原因。
- 保存关联服务记录后，机会自动变为已完成。

### 4.4 服务记录

服务记录用于沉淀真实服务情况。

当前支持：

- 直接新增服务记录。
- 从服务机会进入新增服务记录。
- 自动带入长者和关联机会。
- 保存关联记录后完成对应服务机会。
- 刷新后记录仍保留。

### 4.5 我的

我的页面用于展示个人与机构入口。

当前支持：

- 当前用户信息
- 当前机构信息
- 当前机构身份
- 机构设置入口
- 数据导出备份
- 数据导入恢复
- 隐私与使用边界
- 关于知老

### 4.6 机构设置

机构设置用于维护当前机构基础信息。

当前支持字段：

- 机构名称
- 机构类型
- 所在城市
- 联系人
- 联系电话
- 机构说明

权限规则：

- `org_admin` 可编辑。
- `staff` / `volunteer` 不可编辑。

## 5. 数据与权限现状

### 5.1 当前默认数据源

当前默认数据源仍为浏览器 `localStorage`。

统一 key：

```text
zhilao_app_data
```

当前统一管理的数据包括：

- `currentUser`
- `currentOrganizationId`
- `organizations`
- `organizationMembers`
- `elders`
- `serviceOpportunities`
- `serviceRecords`

### 5.2 当前身份模型

当前不做真实登录注册。

用户身份通过 mock 数据模拟：

- `currentUser`
- `organizationMembers`

用户本身不直接写死角色，用户在机构里的身份来自 `organizationMembers`。

当前角色：

- `org_admin`：机构管理员
- `staff`：服务人员
- `volunteer`：志愿者

### 5.3 当前权限边界

已实现：

- 只有 `org_admin` 可编辑机构设置。
- 只有 `org_admin` 可归档 / 恢复长者。
- 只有 `org_admin` 可看到数据备份入口。

暂未实现：

- 真实账号权限
- 真实登录态
- 多机构切换
- 成员邀请
- 成员审核

## 8. 当前体验与安全边界

体验阶段必须提醒用户：

1. 使用模拟或脱敏信息。
2. 不录入身份证号、详细病历、家庭纠纷、财产情况等敏感信息。
3. 头像照片应使用授权照片、模拟照片或脱敏图片。
4. AI 生成内容仅用于服务辅助，需人工确认后使用。
5. 系统不做医疗诊断。
6. 系统不做心理诊断。
7. 老人信息和服务记录仅用于机构内部服务辅助。

## 9. 验收重点

每个版本迭代后，至少需要验证：

1. `npm run build` 通过。
2. 手机端无横向滚动。
3. PC 端 Sidebar / Header 正常。
4. 首页只展示待处理机会。
5. 归档长者不出现在默认提醒中。
6. 服务机会不能直接变成已完成。
7. 保存关联服务记录后，服务机会自动完成。
8. 关闭服务机会必须选择原因。
9. 头像上传刷新后仍保留。
10. 机构设置刷新后仍保留。
11. 数据导出 / 导入可用。
12. 不配置 Supabase 时应用仍能运行。
